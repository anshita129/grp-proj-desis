Interactive market simulation – user workflow
------------------------------------------------

Audience
This guide ignores implementation details and instead explains what an end user (trader, analyst, demo attendee) actually does when using the interactive market simulation that lives across the `market_simulation` package and the Django `simulation` app.

Three personas
1. "Data Preparer" – ensures market data and basic configuration exist.
2. "Simulation Operator" – starts/stops simulations, advances time, and observes metrics.
3. "Trader" – places trades during the simulation.
One person can play all three roles, but splitting them clarifies the flow.

Environment checkpoints (before any user action)
- Django API is running (e.g., `python manage.py runserver 8003`).
- Static HTML page is served locally (e.g., `python -m http.server 3000` in the `frontend/` folder, then open `http://localhost:3000/interactive.html`).
- Database (`db.sqlite3` by default) already has seeded instruments/users via the bundled management commands.

Step-by-step experience

0. Data Preparer
   - Upload or drop historical CSVs that the simulation will replay.
   - Optionally run `python manage.py simulate_prices` to populate price caches and `python manage.py seed_trading` for sample users/holdings.
   - Confirm that the data list appears in the admin panel or logs (purely a sanity check).

1. Simulation Operator: start a new timeline
   - Open `interactive.html` and hit **Start Simulation**.
   - The page issues `POST /api/simulation/interactive/ { "action": "start" }`.
   - Backend response: a `snapshot_id`, initial cash, current holdings, and headline metrics.
   - The UI stores `snapshot_id` in memory; the operator now sees day 0 plus baseline stats.

2. Simulation Operator: advance market time
   - Press **Advance Day** whenever they want to step forward.
   - Frontend sends `{ "action": "advance", "snapshot_id": "..." }`.
   - Backend loads the saved snapshot, replays the next chunk of prices via the replay engine, updates holdings/PNL, then returns the refreshed summary (new date, prices, metrics).
   - Operator watches prices, P&L sparkline, and trade log update automatically.

3. Trader: place orders while time is paused (or right after advancing)
   - Choose Buy or Sell, type the symbol, quantity, and optional limit price, then submit.
   - Frontend calls `{ "action": "trade", "side": "buy"|"sell", "symbol": "AAPL", "qty": 10, "snapshot_id": "..." }`.
   - Backend validates the order (cash/position check) and routes it through the replay trade queue; a fill adjusts portfolio cash + holdings immediately, and the updated snapshot comes back.
   - The trade log at the bottom of the UI lists each fill, including the simulated execution price.

4. Everyone: inspect portfolio + performance
   - Snapshot payload contains: cash, total equity, unrealized vs realized P&L, open positions, and any strategy-level metrics.
   - Charts/tables on `interactive.html` repaint using that snapshot so even non-technical observers can follow along.
   - For deeper dives, the live dashboard script (`simulate_live_dashboard.py`) can be launched in a terminal; it sits on top of the same snapshot feed and shows metrics scrolling in real time.

5. Optional branches
   - **Comparison mode**: run `run_market_sim.py` or `comparison_runner` separately to benchmark multiple strategies. Results can be pasted/linked during the interactive session to discuss why a strategy is being chosen.
   - **Resume later**: because each step stores snapshots via `SnapshotManager`, the operator can close the browser, reopen later, and send `{ "action": "advance", "snapshot_id": "last_id" }` to continue from that frozen state.

6. Finish session
   - When data runs out or the operator decides to stop, they simply stop advancing. The final snapshot is already persisted.
   - They may export results: download `db.sqlite3`, run `python manage.py dumpdata simulation.Snapshot`, or grab CSV exports from the dashboard (if enabled).

What users see in the UI
- Buttons: Start Simulation, Advance Day, Buy, Sell.
- Data cards: current date, cash, total equity, % return.
- Tables: holdings (symbol, qty, avg price, market value) and trade history.
- Status alerts when API calls fail (e.g., insufficient cash) so the user knows what to fix.

Glossary in everyday language
- Snapshot: a saved picture of your game state (cash, holdings, day, and metrics).
- Advance: move the market clock forward one step and recalc everything.
- Trade: place a pretend order that obeys cash/position rules just like a broker.
- Strategy: optional autopilot logic that can also submit trades; you can watch it act while you place manual ones.

Quick script-based alternative (no UI)
For presenters who prefer terminals:
1. `python run_market_sim.py --strategy basic_momentum --csv data/AAPL.csv`
2. Watch logs for “Tick … order executed … snapshot saved”.
3. Launch `python simulate_live_dashboard.py` to get an ASCII dashboard that mirrors the browser view.

What is *not* required of the user
- They never have to touch Django admin, write code, or restart the server between steps.
- They don’t interact with Celery, Redis, or any message queue; everything is in-process for the demo.

Next user-facing enhancements (nice-to-haves)
- Wizard dialog in the UI to guide a brand-new user through Start → Advance → Trade.
- Inline tutorial cards that explain each metric the first time it appears.
- Ability to bookmark/share a snapshot link (`/interactive.html?snapshot_id=...`).

End of user workflow guide.
