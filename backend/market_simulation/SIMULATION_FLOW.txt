Overview

This document explains, in plain language and with file references, what the project currently implements for the market simulation and simulation subsystems and how a user would flow through them.

Top-level purpose
- The `market_simulation` package provides the engine and supporting components to replay historical price data, run trading strategies against that data, execute simulated trades, capture portfolio snapshots, and compute performance metrics.
- The `simulation` Django app and supporting scripts provide the glue to configure, start, and visualize those simulations (via views, management scripts, and small front-end pages).

High-level user flow (what a user experiences)
1) Prepare / seed data
   - Ensure historical price data is available (CSV or other supported store). The project includes price-store implementations under `market_simulation/replay/` to read CSVs or cached stores.
   - Optionally run management commands in `trading/management/commands` (examples: `simulate_prices.py`, `seed_trading.py`) to populate DB fixtures used by the simulation.

2) Configure a simulation
   - Choose symbol(s), date range, timeframe, initial capital, fees/slippage model, and one or more strategies. Strategy classes live under `market_simulation/strategy/` (see `base.py`).
   - You can configure simulations in code (scripts) or via the lightweight web UI pages (`frontend/interactive.html`, `frontend/extra.html`) if integrated.

3) Start the simulation (entry points)
   - Script entry points and helpers exist at the repository root: `run_market_sim.py` and `simulate_live_dashboard.py`. They initialize the replay engine and, optionally, the live visualization.
   - The core orchestration is handled by `market_simulation/replay/engine.py` (the replay engine). The engine loads a price store, initializes time state and portfolio, and iterates through market ticks.

4) Replay loop and events
   - The engine advances simulated time (using `time_state.py`), pulls next price/tick from a `PriceStore` (e.g., `csv_price_store.py` or `cached_price_store.py`), and emits a market tick event through `market_simulation/events/bus.py`.
   - Registered strategy instances (implementing the interface in `strategy/base.py`) receive tick events and can create orders.

5) Order handling and execution
   - Strategies send orders to the `market_simulation/replay/trade_queue.py`. The trade queue batches/processes them and calls into the trade service (see `trading/service/trade_service.py`) or local execution routines to create fills.
   - Executions update the portfolio via `market_simulation/portfolio/store.py` (positions, cash, realized/unrealized P&L).

6) Snapshotting and metrics
   - Periodically (or on demand) the simulation creates snapshots with `market_simulation/snapshot/manager.py` to save portfolio state and related info. Snapshots are used by front-end views and the live dashboard.
   - Performance metrics (returns, drawdown, Sharpe, etc.) are computed by `market_simulation/metrics/performance.py` and can be stored or presented by the UI.

7) Visualization and finishing
   - `simulate_live_dashboard.py` and front-end pages (`frontend/interactive.html`) display snapshots and metrics, either by polling endpoints or via websocket/event hooks if implemented.
   - At the end of the price data, the engine stops and final snapshots/metrics are persisted. The `comparison_runner.py` can be used to run multiple strategies over the same data and produce comparison reports.

Key files and responsibilities (where to look)
- `market_simulation/replay/engine.py` — Orchestrates the simulation loop: initializes stores, advances time, emits tick events, and coordinates components.
- `market_simulation/replay/csv_price_store.py` — Reads historical ticks/prices from CSV files.
- `market_simulation/replay/cached_price_store.py` — A cached layer for price data optimized to avoid repeated I/O.
- `market_simulation/replay/time_state.py` — Tracks the current simulated timestamp and iteration state.
- `market_simulation/replay/trade_queue.py` — Receives strategy orders and handles sequencing/execution.
- `market_simulation/events/bus.py` — Event bus that publishes ticks, orders, fills, and snapshot events.
- `market_simulation/portfolio/store.py` — In-memory/persistent portfolio model (positions, cash, P&L).
- `market_simulation/snapshot/manager.py` — Saves and restores snapshots for visualization and resuming.
- `market_simulation/strategy/base.py` — Strategy interface and helper utilities.
- `market_simulation/strategy/comparison_runner.py` — Runs multiple strategies for side-by-side comparison and reporting.
- `market_simulation/metrics/performance.py` — Performance calculations.
- `trading/service/trade_service.py` — The component that applies fills and integrates with DB models if the simulation uses the Django `trading` app models.
- `trading/management/commands/*.py` — Helpful management commands for seeding and preparing data.
- `run_market_sim.py`, `simulate_live_dashboard.py` (project root) — Convenience scripts for starting a simulation and the dashboard.

Data & event flow (simple diagram in words)
  Price CSV -> PriceStore -> Replay Engine -> (tick event) -> Strategy -> (order) -> TradeQueue -> TradeService -> Portfolio Store -> Snapshot Manager -> Metrics -> Frontend / DB

Contracts (inputs / outputs / errors)
- Inputs: price source (CSV or cached store), strategy class instance, simulation config (initial cash, fees, slippage, start/end time).
- Outputs: executed trade records (in memory and/or DB), periodic snapshots, final performance metrics, logs.
- Error modes: missing/invalid price data, strategy exceptions (should be caught by engine to avoid crashing the run), out-of-range timestamps, DB write failures when persisting snapshots or trades.

Common edge cases and how the code handles them
- Empty or incomplete CSVs: price store should raise or return no ticks — engine should detect end-of-data and stop.
- Strategy exceptions: tests and runner code typically wrap strategy calls so the engine can continue or stop gracefully and produce an error report.
- Large datasets: `cached_price_store.py` reduces repeated file access and memory pressure; if memory is still too large, stream-based iteration is preferred.
- Replaying resumption: `snapshot/manager.py` can be used to resume from a stored snapshot if implemented.

Where to add or extend
- Add new strategy classes under `market_simulation/strategy/` (implement `on_tick` or the defined callbacks).
- Add new price sources by implementing `PriceStore` with the same interface used in `replay/engine.py`.
- Add visualizations in `frontend/` or expand `simulate_live_dashboard.py` to use websockets for real-time updates.

Tests
- Unit tests for replay and stores are under `market_simulation/tests/` (look at `test_cached_price_store.py`, `test_replay_engine.py`, `test_strategy_comparison.py`). Run them to verify behavior when changing components.

Quick checklist for “How a developer runs a simulation”
1. Make sure historical price files are present or DB is seeded.
2. Choose a strategy and set parameters.
3. Start the engine via `run_market_sim.py` or call the replay engine programmatically from a script or Django view.
4. Watch snapshots/metrics update in `simulate_live_dashboard.py` or the front-end pages.
